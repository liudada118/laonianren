## 2. 失败用例分析

本次测试发现 2 个失败用例，以下是详细分析：

### 2.1 T12 - GET /getColHistory 获取采集历史 (API 超时)

- **问题描述:** 在调用后端 API `/getColHistory` 时，请求在设定的 8 秒超时时间内未能返回响应，导致测试失败。
- **根本原因:** 通过分析后端代码 `server/serialServer.js`，发现此 API 的 SQL 查询语句在 `matrix` 表为空时存在性能问题。查询中使用了 `INNER JOIN` 和复杂的子查询，当数据库中没有任何采集数据时，该查询会陷入长时间等待或死锁，无法正常返回空结果。
- **失败截图:**
  ![T12 失败截图](T12_-_GET__getColHistory_______.png)
  *（截图显示测试在登录页面超时，这是因为测试框架在等待 API 响应时，UI 保持在当前页面）*

### 2.2 T40 - 应用正常关闭 (关闭超时)

- **问题描述:** 在测试结束时，关闭 Electron 应用的 `app.close()` 操作超过了 15 秒仍未完成，导致测试超时失败。
- **根本原因:** 此问题与测试环境的特殊性有关。在开发和测试模式下，主进程会启动一个 Vite 开发服务器作为子进程来提供前端页面。在应用退出时，主进程尝试清理所有子进程，但 Vite 进程的关闭逻辑有时会发生阻塞，导致主进程无法正常退出。
- **影响评估:** 这是一个**测试环境特定**的问题，在用户实际使用的已打包生产版本中**不会出现**，因为生产版本的前端资源是静态文件，不涉及 Vite 开发服务器。

## 3. 问题修复建议

针对以上两个失败用例，我们提出以下修复建议：

1.  **修复 `/getColHistory` API 阻塞问题 (高优先级):**
    - **方案:** 优化 `server/serialServer.js` 中 `getColHistory` 路由的 SQL 查询逻辑。
    - **建议:** 在执行复杂的 `INNER JOIN` 查询之前，首先检查 `matrix` 表是否为空。如果为空，则直接返回一个空的数组结果，避免执行可能导致阻塞的查询。

    ```javascript
    // 伪代码修复建议
    app.get('/getColHistory', async (req, res) => {
      // 1. 先检查表是否为空
      currentDb.get("SELECT COUNT(*) as total FROM matrix", (err, row) => {
        if (err || row.total === 0) {
          return res.json(new HttpResult(0, [], 'success'));
        }
        // 2. 如果不为空，再执行原有的复杂查询
        const selectQuery = `...`; // 原有的复杂 SQL
        currentDb.all(selectQuery, [0, 500], (err, rows) => {
          // ... 返回查询结果
        });
      });
    });
    ```

2.  **优化测试脚本中的应用关闭逻辑 (中优先级):**
    - **方案:** 虽然这是测试环境问题，但为了提高测试的稳定性和可靠性，可以改进测试脚本的关闭逻辑。
    - **建议:** 在调用 `app.close()` 后，增加一个强制杀死相关进程的步骤，确保测试环境能够干净利落地结束。

    ```javascript
    // 伪代码修复建议 (in e2e_test.js)
    test: withTimeout(async (ctx) => {
      await ctx.app.close();
      await wait(2000);
      // 如果进程仍然存在，则强制杀死
      if (ctx.app.process() && !ctx.app.process().killed) {
        process.kill(ctx.app.process().pid, 'SIGKILL');
      }
    })
    ```
