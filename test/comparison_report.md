# 脚本测试 vs. 完全点击测试 对比分析报告

**版本**: 1.0
**日期**: 2026-03-03
**作者**: Manus AI

## 1. 测试背景

本次测试旨在对比两种不同的自动化测试方法在“老年人筛查系统 MAC”项目中的表现和发现问题的能力：

1.  **脚本测试 (v3)**: 主要通过直接调用后端 API 和注入 JavaScript 来执行测试流程，绕过了部分前端 UI 交互。
2.  **完全点击测试 (v5)**: 严格模拟真实用户的操作路径，所有交互均通过在 UI 界面上查找并点击按钮、链接等元素来完成。

通过对比这两种方法的测试结果，我们可以评估完全模拟用户操作的必要性，并发现仅在真实交互场景下才会暴露的 Bug。

## 2. 测试结果对比

两种测试方法的通过率和发现问题的类型存在显著差异，具体如下表所示：

| 对比项 | 脚本测试 (v3) | 完全点击测试 (v5) | 差异分析 |
| :--- | :--- | :--- | :--- |
| **通过率** | **95%** (22/23) | **100%** (23/23) | 点击测试在修复选择器后通过率更高，说明 UI 元素本身是可交互的。 |
| **发现的 Bug 数量** | 2 个 (UX) | 4 个 (2 UX, 2 功能性) | 点击测试发现了更多、更严重的 Bug。 |
| **Bug 类型** | UX Bug (状态反馈不一致) | **功能性 Bug** (路由失败、数据保存失败) + UX Bug | 点击测试能发现由真实交互触发的功能性 Bug。 |
| **测试真实性** | 低 (绕过 UI) | **高** (完全模拟用户) | 点击测试更贴近真实用户的使用场景。 |

## 3. 关键发现：点击测试暴露的严重 Bug

完全点击测试成功发现了多个脚本测试未能覆盖的严重问题，这些问题直接影响了应用的核心功能和用户体验。

### 3.1 功能性 Bug

1.  **返回 Dashboard 失败 (高优先级)**
    -   **现象**: 在任何评估页面（如握力、起坐）完成后，点击“返回首页”按钮，页面 URL 变化但视图不更新，仍然停留在评估页面。
    -   **根因**: 这是前端路由的一个严重 Bug。`navigate('/dashboard')` 调用可能被某个状态或副作用 (side effect) 阻止了，导致页面无法正确跳转。
    -   **影响**: 用户无法正常返回主页，核心流程中断。

2.  **历史记录不完整 (高优先级)**
    -   **现象**: 即使在 UI 上完整地执行了握力和起坐两次评估，历史记录页面只显示握力评估为“已完成”，完成度为 1/4。
    -   **根因**: 起坐评估完成后，调用 `completeAssessment` 函数可能失败或被跳过，导致评估结果没有被正确保存到后端数据库。
    -   **影响**: 用户评估数据丢失，历史记录功能不可靠。

### 3.2 用户体验 (UX) Bug

1.  **评估按钮与文字分离**
    -   **现象**: 在所有评估页面中，“开始采集”和“结束采集”的圆形按钮与其下方的文字标签在 HTML 结构上是分离的。`onClick` 事件绑定在按钮上，而文字标签是纯 `<span>`。
    -   **影响**: 用户可能习惯性地点击文字区域，但这不会触发任何操作，导致困惑和操作失败。

2.  **历史记录中报告无法查看**
    -   **现象**: 在历史记录详情中点击“查看报告”按钮，页面跳转后显示“没有报告数据”。
    -   **根因**: 这与脚本测试中发现的问题一致，是由于前后端对报告数据的格式定义不匹配导致的。

## 4. 结论与建议

对比结果明确表明，**完全模拟用户点击操作的测试方法远优于脚本测试**。尽管脚本测试可以快速验证后端 API 的连通性和基本功能，但它完全无法发现由前端状态管理、路由和真实用户交互流触发的严重 Bug。

**核心结论**: **对于带有复杂前端交互的应用，必须采用完全模拟用户点击的端到端测试策略。**

### 建议

1.  **修复高优先级 Bug**: 立即修复“返回 Dashboard 失败”和“历史记录不完整”这两个功能性 Bug。
2.  **改进 UX**: 统一评估页面的采集按钮结构，将 `onClick` 事件绑定到包含文字和图标的整个可点击区域，提升用户体验。
3.  **采纳点击测试**: 将本次编写的 `e2e_full_click_test.js` 作为项目回归测试的基础，在未来的开发迭代中持续运行和维护，确保核心用户流程的稳定性。
